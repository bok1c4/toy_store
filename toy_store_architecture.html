§ 01 / TOPOLOGY

Five services,
one ingress, one wire.

Everything the user sees is delivered through Nginx on port 80. The four backing services — nginx, frontend, backend, db, redis — communicate over the internal Docker network. Only Nginx publishes a host port. The backend additionally calls one external service: toy.pequla.com/api, which owns the toy catalog.

§ 02 / NGINX

The gatekeeper:
routing & rate limits.

Nginx is the only public surface. It does three jobs: it splits traffic between the Next.js app and the Go API, it applies two distinct rate-limit zones, and it lets /health through without throttling so Docker can run its 10-second healthcheck. Decisions happen before any request reaches the Go process.

Two zones, two intents

The strict zone exists to defeat brute-force on /api/auth/login. The general zone exists to throttle scrapers without bothering legitimate users.

auth_limit
10 req/min, burst=5, 10MB zone, keyed on client IP. Applied to /api/auth/.
api_limit
200 req/min, burst=50, 10MB zone, keyed on client IP. Applied to all other /api/ traffic.
/health
Exact match. Bypasses both zones so Docker's healthcheck (every 10s) is never throttled.
/api/webhook/stripe
Falls under api_limit (no special rule). README's "no special rate limit" reads accurately.

Why at the edge

Putting the limit at Nginx — not in Go middleware — means a flood gets stopped before the Go runtime allocates a goroutine, parses the body, or touches the database. The cost of a rejected request stays as low as possible.

⚑ Note · Belt-and-suspenders

The backend also wraps POST /api/auth/register and POST /api/auth/login with an in-process ulule/limiter at 10/min. So auth endpoints have rate-limiting at both the Nginx and Gin layers. router.go:62-67, 70-71

§ 04 / AUTHENTICATION

JWT, but revocable.

Two tokens. The access token (15 minutes) goes in the Authorization header on every protected call. The refresh token (7 days) is held in Redis under refresh:<userID>:<jti> and used only to mint a new pair. Storing the refresh token server-side is what makes logout actually work — without it, JWT is unrevocable until expiry.

What Redis buys you

Storing the refresh token in Redis turns a stateless JWT into a server-checkable session. On logout, the key is deleted; future refresh attempts return 401 before a new pair is ever minted. The user can still send the old access token until its 15-minute window closes, but cannot extend their session.

Signing
HS256 · single shared secret JWT_SECRET
Claims

RBAC by route group, not by handler

Admin protection is applied at the router group level, not on each handler:

adminGroup := r.Group("/api/admin")
adminGroup.Use(authMiddleware.RequireAuth())
adminGroup.Use(authMiddleware.RequireAdmin())
{
    adminGroup.GET("/users", ...)
    adminGroup.GET("/orders", ...)
    // every new endpoint inherits both
}

The implication: a developer who adds a new admin endpoint cannot accidentally forget to protect it. Security by default.

✓ Refresh TTL is now driven by env

The refresh token's Redis TTL was previously hardcoded 168 * time.Hour. It now reads from JWTManager.RefreshTTL(), which is itself populated from JWT_REFRESH_TTL. The JWT exp claim and the Redis revocation key now expire at the same instant, regardless of the env value.

verified · TTL on refresh:<uid>:<jti> observed at 604799s (= 168h - 1s rounding) · matches JWT exp

§ 05 / DATABASE

Five tables.
One snapshot trick.

Postgres stores users, orders, order line items, cart, and wishlist. The catalog is not stored — toys come from an external API. So when an order is placed, line items capture a price/name/image snapshot. If the external API changes a price tomorrow, today's order history doesn't lie.

The snapshot pattern

The catalog lives outside this system. Prices change. Items disappear. Orders, however, must be immutable historical records.

Solution: order_items stores toy_name, toy_image_url, and price_at_purchase as denormalised columns. toy_id is INTEGER with no foreign key — you can't FK to a system you don't own.

cart_items uses the same approach via *_cache columns, for fast cart rendering without round-tripping the external API.

§ 06 / CACHE

Cache-aside,
with graceful degradation.

Whenever a toy is requested, the backend checks Redis first. On hit: ~1ms response. On miss: HTTP call to toy.pequla.com, write the result back to Redis, return. If Redis is unavailable, the write fails — the backend logs a warning and returns the data anyway. The cache is an accelerator, not a hard dependency.

§ 07 / PAYMENT

Stripe, or
mock. The key decides.

At startup the backend checks STRIPE_SECRET_KEY. If it begins with sk_, real Stripe (test mode) is on: a two-step flow of intent → confirm means the card never reaches our server. If the key is empty or anything else, the system falls back to a mock flow and creates an order immediately. One code path; one runtime branch.

Why two steps

The intent/confirm split keeps the credit card off our server entirely. Stripe.js encrypts and sends the card to Stripe's PCI-compliant infrastructure. Our backend only sees a payment_intent_id and queries Stripe for its status. We never store, log, or proxy a card number.

Currency
"rsd" — Serbian dinar
Amount unit
cart total × 100 (Stripe's smallest-unit convention)
Test card
4242 4242 4242 4242 · any future date · any 3-digit CVC
Order tx
INSERT orders → INSERT items → DELETE cart, all-or-nothing

The webhook is observability-only

The Stripe webhook endpoint POST /api/webhook/stripe verifies the HMAC-SHA256 signature against STRIPE_WEBHOOK_SECRET, then logs payment_intent.succeeded / payment_intent.payment_failed. It does not mutate any DB row.

✓ Webhook is observability-only by design

The architectural decision is documented: the order's payment_status is committed in-band at confirm-time, before the webhook fires. The webhook exists to log async Stripe events for monitoring, not to drive state transitions. See § 10 for the full posture.

§ 08 / FRONTEND

One server page,
sixteen client.

The Next.js app uses the App Router, but in practice almost every page is a Client Component. The home page (app/page.tsx) is the lone Server Component — it fetches the catalog server-side over the Docker network. State lives in Zustand. Tokens live in both localStorage and cookies. The Axios interceptor refreshes silently on 401.

§ 10 / WEBHOOK POSTURE

The webhook
is a witness, not a judge.

After remediation, the Stripe webhook's role is explicit. Order state is decided in-band at confirm-time, before any webhook event arrives. The webhook exists to log payment_intent.succeeded and payment_intent.payment_failed for monitoring, audit, and future-state reconciliation if it's ever needed. It does not write to the database. This is a deliberate choice; the trade-offs are below.

Why this design

The trade-off has a name in distributed-systems literature: synchronous decision, asynchronous observation. The user gets a fast, atomic answer. The webhook gives the operator a parallel record for monitoring and audit. The two paths can diverge — and if they ever do, that divergence is the bug, not the design.

Driving state from the webhook would mean accepting "order created but unpaid" as a transient state, with all the UI and recovery logic that implies. The current scope (academic project, mock-mode default) does not warrant that complexity.
